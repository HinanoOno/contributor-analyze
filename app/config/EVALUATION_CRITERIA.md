# 評価基準リスト

### 1. リーダーシップ領域
- **説明:** チームをまとめ、目標に向かって導く能力をもつこと。ビジョンを示し、部下を育成する力が評価される領域である。
- **GitHub上での行動例:**
  - 複数のリポジトリやチームにまたがる横断的な課題を解決するためのIssueを起票し、議論を主導している。
  - プロジェクト全体のアーキテクチャや技術的負債に関する議論を提起し、改善案をPRで提案している。
  - 障害発生時に、自チームの担当範囲外であっても原因調査に協力し、解決策をコメントしている。

### 評価レベル
- **Mentor (レベル4):** ロールモデル。事業や顧客のために解決すべき本質的な課題を発見し、関係者にとって厳しい要求でも周囲を巻き込み解決に向けて行動できる。経営に近い視座を持つ。
- **Very good (レベル3):** 十分な成果を出している。仕事の範囲を自チームと関連ステークホルダーまで広げて捉えている。
- **Nice try (レベル2):** 頑張っている。仕事の範囲をチームとして捉え行動するが、事業全体への関心は薄いことがある。
- **Standard (レベル1):** 期待通り。仕事の範囲を主に個人として捉えているが、チームへの貢献を意識し始めている。視座を上げるには、上長やメンバーからのサポートが必要な段階。
- **Neutral (レベル0):** 影響なし。指示された個人のタスクは遂行するが、それ以上の主体的な動きやチームへの働きかけは見られない。良くも悪くもチームへの影響は限定的。
- **Needs Improvement (レベル-1):** 期待以下。チームや事業の目標に無関心で、担当範囲外の課題を「自分には関係ない」と他人事として扱う傾向が強い。受け身の姿勢がチームの生産性を僅かに下げている。

### 2. チームワーク領域
- **説明:** 個人の成果だけでなく、チーム全体の目標達成を重視し、他者と積極的に協力・連携する姿勢。メンバーの意見を尊重し、必要なサポートを提供・要求しながら、知識や情報をオープンに共有することで、1+1を2以上にする相乗効果を生み出す力。
- **GitHub上での行動例:**
  - 他のメンバーのPRを積極的にレビューし、コードの改善提案だけでなく、仕様の抜け漏れやドキュメントの分かりやすさについてもフィードバックしている。
  - 他のメンバーが担当するIssueで困っている際に、自ら調査を手伝ったり、関連情報や解決策のヒントをコメントで提供している。
  - 自分が得た知見や便利なツールについて、IssueやWikiなどで積極的に共有し、チーム全体の生産性向上に貢献している。

### 評価レベル
- **Mentor (レベル4):** ロールモデル。チームの心理的安全性を高め、メンバー間の対立を建設的な議論に導くことができる。部署やチームの垣根を越えたコラボレーションを自ら生み出し、組織全体の成果に貢献する。
- **Very good (レベル3):** 十分な成果を出している。チームの目標達成のために、自身のタスク範囲を超えて積極的に他者をサポートし、知識共有を率先して行う。
- **Nice try (レベル2):** 頑張っている。チームへの貢献意欲は高いが、サポートが受け身になったり、情報共有が部分的に留まったりすることがある。
- **Standard (レベル1):** 期待通り。求められれば他者の作業に協力し、チーム内の議論にも参加する。自発的な働きかけは少ないが、孤立することはない。
- **Neutral (レベル0):** 影響なし。主に個人のタスクに集中し、チームへの関与が薄い。協力を求めることも、提供することもほとんどない。
- **Needs Improvement (レベル-1):** 期待以下。知識を囲い込んだり、他者への協力に非協力的であったりする。批判的な態度が多く、チームの雰囲気を悪化させることがある。

### 3. 問題解決領域
- **説明:** 発生した問題に対し、根本原因を特定し、構造的に解決策を立案・実行する能力。場当たり的な対応ではなく、再現性のあるアプローチで課題に取り組む姿勢を指す。
- **GitHub上での行動例:**
  - 複雑なバグの修正PRで、原因調査のプロセス（仮説→検証）と、なぜその修正方法を選んだのかを明確に説明している。
  - 実装方針が複数考えられる課題について、各案のメリット・デメリットをIssueコメントで提示し、議論のたたき台を作成している。
  - Issueの報告時に、バグの再現手順やエラーログ、期待される挙動を具体的に記述し、他の人が調査しやすいようにしている。

### 評価レベル
- **Mentor (レベル4):** ロールモデル。潜在的な問題を予見し、発生する前に手を打つことができる。チーム全体の課題解決能力を引き上げるための指導や仕組み作りに貢献する。
- **Very good (レベル3):** 十分な成果を出している。複雑な問題に対しても、自ら原因を特定し、複数の解決策を比較検討した上で、最適な打ち手を実行できる。
- **Nice try (レベル2):** 頑張っている。問題解決に取り組む姿勢はあるが、原因の特定が表層的であったり、解決策の検討が不十分であったりすることがある。
- **Standard (レベル1):** 期待通り。明確に指示された問題については解決できるが、曖昧な問題や未知の課題に対しては、自ら調査・解決するのに苦労する。
- **Neutral (レベル0):** 影響なし。指示されたタスクはこなせるが、問題が発生しても自ら解決しようとせず、誰かの指示を待つ姿勢が目立つ。
- **Needs Improvement (レベル-1):** 期待以下。問題に直面した際に、十分な調査を行ずに他者に助けを求めたり、場当たり的な修正で新たなバグを生み出したりすることがある。

### 4. コミュニケーション領域
- **説明:** 業務の意義や状況について、事実と意見・推測を分けて、相手に分かりやすく伝える能力。情報の透明性を保ち、チームの円滑な意思決定と連携を促進する。
- **GitHub上での行動例:**
  - PRの説明文で、変更の「背景（事実）」と、それに対する「自分の考え（意見）」を明確に分けて記述している。
  - Issueでの議論において、決定事項とその理由を客観的な事実に基づいて要約・記録している。
  - 複雑な議論を「論点A」「論点B」のように整理し、議論を前に進めるコメントをしている。

### 評価レベル
- **Mentor (レベル4):** ロールモデル。発言しにくいメンバーが発言しやすい環境を作れる。相手の理解度や状況に合わせて伝え方を工夫し、チーム全体のコミュニケーション品質を高める。
- **Very good (レベル3):** 十分な成果を出している。事実と考えを分け、常に相手に伝わっているか確認しながら話すため、誰も置き去りにしないコミュニケーションができる。
- **Nice try (レベル2):** 頑張っている。説明を求められた際に言語化しようと努め、事実と考えを分けて話すことができるが、表現が冗長になったり、要点が不明確なことがある。
- **Standard (レベル1):** 期待通り。説明が不十分、または背景の咀嚼なく表層的な事実のみを伝えることがあるが、指摘されれば説明し直そうと努める。
- **Neutral (レベル0):** 影響なし。説明を求められても最低限の情報しか伝えず、事実と意見の区別を意識していない。議論への貢献度が低い。
- **Needs Improvement (レベル-1):** 期待以下。思い込みや推測を事実であるかのように話すことがあり、周囲に誤解や混乱を生じさせる。

### 5. 適応力領域
- **説明:** 仕様変更や優先順位の変化、予期せぬトラブルなど、状況の変化に柔軟に対応する能力。既存のやり方に固執せず、常に最適な方法を模索し、変化を前向きに受け入れる姿勢。
- **GitHub上での行動例:**
  - PRのレビューで仕様変更の指摘を受けた際に、議論を通じて建設的に受け入れ、迅速に修正対応している。
  - プロジェクトの優先順位が変更された際に、担当していたタスクを適切に中断・整理し、新しいタスクに素早く切り替えている。
  - 新しい技術やライブラリを検証するため、小規模なPoC（概念実証）リポジトリを作成し、その結果をチームに共有している。

### 評価レベル
- **Mentor (レベル4):** ロールモデル。予期せぬ変化や困難な状況を楽しみ、チームが変化に対応できるよう積極的にサポートし、変化を成長の機会に変えることができる。
- **Very good (レベル3):** 十分な成果を出している。急な仕様変更やトラブルにも動じず、冷静に状況を分析し、前向きに対応策を検討・実行できる。
- **Nice try (レベル2):** 頑張っている。状況の変化に対応しようと努力するが、時に不満や戸惑いを見せることがあり、気持ちの切り替えに時間がかかることがある。
- **Standard (レベル1):** 期待通り。指示された変更には対応するが、自ら変化を提案したり、積極的に新しいやり方を取り入れたりすることは少ない。安定を好む傾向がある。
- **Neutral (レベル0):** 影響なし。常に従来通りのやり方を選択し、新しい方法を試そうとしない。現状維持を好み、変化を嫌う。
- **Needs Improvement (レベル-1):** 期待以下。予期せぬ変更に対して抵抗感が強く、不満を表明したり、対応が遅れたりすることで、チームの生産性に悪影響を与えることがある。

### 6. 継続的な学習・自己改善領域
- **説明:** 他者からのフィードバックを素直に受け入れ、常に自身の知識やスキルを向上させようとする姿勢。現状に満足せず、新しい技術やベストプラクティスを積極的に学び、チームやプロダクトに還元する力。
- **GitHub上での行動例:**
  - レビューで修正依頼を受けた際に、「ご指摘ありがとうございます！勉強になります」と感謝を伝え、指摘内容を真摯に反映している。
  - 自身が学んだ新しい技術や知識について、Wikiや勉強会などを通じてチームに共有している。
  - 受けたフィードバックを基に、自身のコーディングスタイルや設計アプローチを継続的に改善している。

### 評価レベル
- **Mentor (レベル4):** ロールモデル。自身の学習だけでなく、チーム全体の学習意欲を高めるための文化醸成や仕組み作りに貢献する。他のメンバーの成長を積極的に支援する。
- **Very good (レベル3):** 十分な成果を出している。ネガティブフィードバックを歓迎する姿勢で受け入れ、冷静に質問し、感謝を伝えることができる。学んだことを積極的に業務に活かす。
- **Nice try (レベル2):** 頑張っている。フィードバックを受け入れようと努めるが、時折、防御的な反応を見せることがある。学習意欲はあるが、業務への還元が不十分なことがある。
- **Standard (レベル1):** 期待通り。フィードバックに対し、言い訳をすることがあるが、最終的には指摘を受け入れ、行動を修正しようとする。学習は受け身であることが多い。
- **Neutral (レベル0):** 影響なし。フィードバックを受け入れても、その後の行動に特に変化が見られない。聞き流している状態が多い。
- **Needs Improvement (レベル-1):** 期待以下。フィードバックに対して不満な態度を示し、内容を理解しようとしない。同じ指摘を繰り返し受けることがある。
